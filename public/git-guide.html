01 — Conceptos base

¿Qué es un repositorio?

Antes de tocar la terminal, entendamos qué es todo esto. Sin jerga técnica.

📁
En Figma / Drive
Archivo de proyecto
Guardás versiones, compartís con el equipo, todos pueden editar
→
🗃️
En código
Repositorio (Repo)
Carpeta especial que guarda TODO el historial de cambios del proyecto
📸

Git = Historial de fotos

Git saca una "foto" de tu proyecto cada vez que vos se lo pedís. Podés volver a cualquier foto anterior en cualquier momento.

Git
☁️

GitHub = Google Drive del código

GitHub es donde subís esas fotos a la nube. El equipo entero puede verlas, bajarlas y trabajar sobre ellas.

GitHub
💻

Local vs. Remoto

Local = en tu computadora. Remoto = en GitHub. Siempre sincronizás entre ambos con push y pull.

Concepto
💡 La analogía perfecta Imaginá que Git es como el historial de versiones de Figma, pero para carpetas enteras de código. GitHub es el servidor donde vive ese historial compartido.
02 — Primeros pasos

Crear un nuevo repositorio

Dos formas: desde GitHub (recomendada para designers) o desde la terminal.

🖱️ Opción A — Desde GitHub (más fácil)

1

Ir a github.com

Iniciá sesión y hacé click en el botón + arriba a la derecha → "New repository".

2

Ponerle nombre

Elegí un nombre sin espacios. Ej: mi-sistema-de-diseno. Activá "Add a README file".

3

¡Listo!

Tu repo ya existe en la nube. Ahora lo podés clonar a tu computadora (ver sección 03).

⌨️ Opción B — Desde la terminal

TerminalCrear repo local
# 1. Ir a la carpeta del proyecto cd /ruta/a/mi-proyecto # 2. Inicializar git git init # 3. Conectar con GitHub git remote add origin https://github.com/user/repo.git # 4. Primer push git push -u origin main
✅ Tip para designers Si usás GitHub Desktop, podés hacer todo esto con una interfaz visual sin tocar la terminal.
03 — Traer el código a tu máquina

Clonar un repositorio

Clonar = descargar una copia completa del proyecto (con todo su historial) a tu computadora.

1

Copiar la URL del repo

En GitHub, entrá al repositorio → botón verde "Code" → copiá la URL HTTPS: https://github.com/equipo/proyecto.git

2

Elegir dónde guardar

En la terminal, navegá a la carpeta donde querés que viva el proyecto: cd ~/Documents/proyectos

3

Ejecutar el comando clone

Escribí git clone URL. Git crea una carpeta con todo el proyecto adentro automáticamente.

Terminalgit clone
# Clonar en la carpeta actual git clone https://github.com/equipo/proyecto.git # Clonar con un nombre personalizado git clone https://github.com/equipo/proyecto.git mi-nombre-local # Entrar al proyecto clonado cd proyecto # Ver el estado actual git status

📂 Tengo una carpeta ya creada — ¿Cómo la conecto a un repo?

Este es el caso donde ya trabajaste en local y ahora querés subir todo a GitHub.

TerminalConectar carpeta existente a GitHub
# 1. Entrar a tu carpeta existente cd /ruta/a/mi-carpeta-existente # 2. Inicializar git en ella git init # 3. Agregar todos los archivos git add . # 4. Primer commit git commit -m "feat: subida inicial del proyecto" # 5. Conectar con el repo de GitHub (crealo antes en github.com) git remote add origin https://github.com/tuuser/tu-repo.git # 6. Renombrar rama a main (estándar actual) git branch -M main # 7. Subir todo git push -u origin main
04 — El flujo del día a día

Hacer un commit

Un commit es como "guardar" en videojuegos — un punto de control al que siempre podés volver.

💾
En Figma
Guardar versión
"Versión para cliente — 20 enero"
≈
📦
En Git
git commit
"feat: nuevo botón primario en componente Card"
1

git status — Ver qué cambió

Antes de guardar, chequeá qué archivos modificaste. En rojo = sin seguimiento. En verde = listo para commit.

2

git add — Elegir qué guardar

git add . agrega TODO. git add archivo.js agrega solo ese archivo.

3

git commit — Sacar la foto

El mensaje describe QUÉ cambiaste. Sé específico: "fix: color de hover en botones" es mejor que "cambios".

4

git push — Subir a la nube

Sube tus commits locales a GitHub. Sin push, los cambios solo existen en tu computadora.

TerminalFlujo completo de commit
# Ver qué cambió git status # Agregar todo al próximo commit git add . # O solo ciertos archivos git add src/components/Button.jsx src/styles/button.css # Crear el commit con mensaje descriptivo git commit -m "feat: rediseño de botón primario con nuevo radio y sombra" # Subir a GitHub git push # Ver historial de commits git log --oneline
📝 Cómo escribir buenos mensajes de commit Usá prefijos: feat: (nueva funcionalidad), fix: (corrección), style: (cambios visuales), docs: (documentación). Ej: style: actualizar tokens de color a nueva paleta
05 — Orientarse en el código

Estructura de carpetas

Cada tecnología tiene su convención. Acá te explicamos qué significa cada carpeta y por qué existe.

📁 mi-proyecto/ ├── 📄 .gitignore # archivos que git ignora ├── 📄 package.json # lista de dependencias ├── 📄 README.md # documentación │ ├── 📁 public/ # archivos estáticos │ ├── favicon.ico │ └── logo.svg │ ├── 📁 src/ # ← acá vivís vos │ ├── 📁 components/ # piezas reutilizables │ │ ├── Button.jsx │ │ ├── Card.jsx │ │ └── Modal.jsx │ │ │ ├── 📁 pages/ # cada pantalla │ │ ├── Home.jsx │ │ └── About.jsx │ │ │ ├── 📁 styles/ # CSS y tokens │ │ ├── tokens.css # colores, tipografía │ │ └── globals.css │ │ │ └── 📁 assets/ # imágenes, íconos, fonts │ └── 📁 node_modules/ # NO tocar, auto-generado

📁 src/components/

Las "piezas" del diseño hecho código. Cada componente de Figma suele tener su archivo acá. Acá es donde más vas a trabajar.

📁 src/styles/ o tokens/

Acá viven los design tokens: colores, tipografías, espaciados. Si cambiás algo acá, afecta a todo el proyecto.

📁 public/

Imágenes y assets que van directamente al navegador. Logo, favicon, PDFs, etc.

📁 node_modules/

Auto-generado. Nunca lo toques ni lo subas a git. Son las dependencias instaladas — se regeneran solas.

⚠️ No editar
📁 mi-proyecto-vue/ ├── 📄 nuxt.config.ts # configuración global ├── 📄 package.json │ ├── 📁 pages/ # rutas automáticas │ ├── index.vue # → / (home) │ └── about.vue # → /about │ ├── 📁 components/ │ ├── 📁 ui/ # componentes base │ └── 📁 layout/ # header, footer │ ├── 📁 assets/ │ ├── css/tokens.css # design tokens │ └── 📁 images/ │ └── 📁 public/ # archivos estáticos

📁 pages/

En Nuxt/Vue, cada archivo .vue en esta carpeta automáticamente se convierte en una página. index.vue es el home.

📄 *.vue (archivos)

Un archivo .vue tiene todo junto: estructura HTML, estilos CSS y lógica. Como un componente de Figma con todo adentro.

📁 composables/

Funciones reutilizables entre páginas. Como los estilos compartidos en Figma, pero para comportamiento.

📁 cualquier-proyecto/ │ ├── 📄 .gitignore # archivos excluidos de git ├── 📄 README.md # "qué es este proyecto" ├── 📄 CHANGELOG.md # historial de cambios │ ├── 📁 src/ # código fuente principal ├── 📁 dist/ # versión compilada (NO editar) ├── 📁 docs/ # documentación ├── 📁 tests/ # pruebas automáticas │ └── 📁 .github/ # configuración de GitHub └── 📁 workflows/ # automatizaciones CI/CD

📄 .gitignore

Lista de archivos que git NO rastrea: node_modules/, archivos con contraseñas (.env), temporales del sistema.

📁 dist/ o build/

Versión "compilada" lista para producción. Nunca edites archivos acá — se regeneran automáticamente.

Solo lectura

📄 README.md

La "portada" del proyecto. Explica qué es, cómo instalarlo y cómo usarlo. Siempre léelo primero.

⚠️ Archivos que nunca debés commitear .env (contraseñas y API keys), node_modules/ (pesado, auto-generado), archivos temporales del sistema (.DS_Store en Mac). Asegurate de que estén en el .gitignore.
06 — Trabajo en equipo

Git, branches y ramas

Las branches (ramas) son copias paralelas del proyecto donde podés trabajar sin afectar la versión principal.

🌿
Analogía
Duplicar archivo Figma
Trabajás sobre la copia, probás sin miedo, y cuando está bien lo unís al original
≈
🌱
En Git
Branch (rama)
Cada feature o cambio vive en su rama hasta estar listo para mergear a main
Visualización de ramas — hover para ver mensajes →
main
producción estable
feature/
nueva funcionalidad
fix/
corrección rápida
main — estable feature/ — nuevas funciones fix/ — correcciones
TerminalTrabajar con branches
# Ver en qué rama estás git branch # Crear nueva rama y moverse a ella git checkout -b feature/nuevo-navbar # Moverse a una rama existente git checkout main # Traer los últimos cambios de main git pull origin main # Unir una rama a main (merge) git merge feature/nuevo-navbar # Subir tu rama a GitHub git push origin feature/nuevo-navbar

🔀 Pull Request (PR)

Cuando terminás tu rama, abrís un Pull Request en GitHub para que el equipo la revise antes de mergear a main.

Review

⚡ Merge

Unir dos ramas. Cuando el PR es aprobado, los cambios de la rama pasan a main y están en producción.

Integrar

💥 Conflictos

Cuando dos personas editaron el mismo archivo, git no sabe cuál mantener. El equipo de dev lo resuelve — avisales.

Pedir ayuda
🌿 Convención de nombres de ramas main = producción | develop = desarrollo activo | feature/nombre = nueva funcionalidad | fix/nombre = corrección | hotfix/nombre = urgente en producción
07 — Publicar al mundo

Deploy: de código a internet

Deploy = publicar el proyecto. Pasar de "funciona en mi computadora" a "funciona para todos en internet".

🖨️
Analogía
Exportar a PDF
Convertís tu archivo de Figma en algo que otros pueden abrir sin Figma
≈
🚀
En código
Deploy
Convertís el código fuente en un sitio web que cualquier navegador puede abrir
▲

Vercel

El más popular para React/Next. Conectás tu repo de GitHub y cada push a main se publica automáticamente. Gratis para proyectos pequeños.

Recomendado
🌐

Netlify

Similar a Vercel. Preview automático por cada PR — ¡ideal para designers! Podés ver el resultado real antes del merge.

Gratis, directo desde GitHub. Perfecto para portfolios, documentación y proyectos simples sin backend.

08 — Design Systems & componentes

Storybook

Storybook es un catálogo interactivo de todos los componentes del sistema de diseño, pero en código real y en el navegador.

📚
En Figma
Página de componentes
Donde listás todos tus components con sus variantes y estados
≈
📖
En código
Storybook
Igual pero en código real: cada componente con sus props, estados e interacciones

👀

Para designers: inspección

Podés inspeccionar cualquier componente en el estado que quieras: hover, disabled, loading. Sin esperar que el dev te lo muestre.

📐

Stories = estados del componente

Cada "story" es una variante o estado del componente. Como las variants de Figma, pero en código real que interactúa de verdad.

🔗

Fuente de verdad compartida

Storybook puede conectarse al diseño de Figma. Es el puente entre el design system y la implementación en código.

TerminalCorrer Storybook
# Instalar Storybook en un proyecto existente npx storybook@latest init # Correr Storybook (abre en el navegador) npm run storybook # Build para deployar npm run build-storybook # Se abre en: http://localhost:6006
JavaScript / JSXButton.stories.jsx — cómo se ve una story
// Cada archivo .stories.jsx define las variantes de un componente export default { title: 'Atoms/Button', // ← dónde aparece en el sidebar component: Button, }; // Cada "story" es una variante (como en Figma) export const Primary = { args: { variant: 'primary', label: 'Confirmar' }, }; export const Disabled = { args: { variant: 'primary', label: 'Confirmar', disabled: true }, };
💡 Chromatic — Storybook en la nube Chromatic es el "GitHub para Storybook". Guarda capturas de cada componente y alerta si algo cambió visualmente entre commits. Como un diff visual automático. Muy usado por equipos de design systems.

Cheat Sheet esencial

COMANDOS BÁSICOS
git init → crear repo local git clone [URL] → copiar repo git status → ver qué cambió git add . → preparar cambios git commit -m "msg" → guardar snapshot git push → subir a GitHub git pull → bajar cambios git log --oneline → ver historial
BRANCHES
git branch → ver ramas git checkout -b [rama] → crear y moverse git checkout main → volver a main git merge [rama] → unir rama git push origin [rama] → subir rama git stash → guardar sin commit git stash pop → recuperar stash
GLOSARIO PARA DESIGNERS
Repository (Repo)
La carpeta del proyecto con todo su historial de cambios
Commit
Un "snapshot" del proyecto en un momento. Como guardar versión en Figma.
Branch / Rama
Copia paralela del proyecto para trabajar sin afectar lo estable
Merge
Unir dos ramas. Incorporar los cambios de una rama a otra.
Pull Request (PR)
Solicitud de revisión antes de mergear. Revisión de pares en equipo.
Deploy
Publicar el proyecto. Pasar el código a internet para que todos lo vean.
main / master
La rama principal. Lo que está acá va a producción.
Storybook
Catálogo visual e interactivo de todos los componentes del sistema.
